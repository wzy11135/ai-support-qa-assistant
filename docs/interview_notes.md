# Interview Notes

## 1 分钟项目介绍

AI Support QA Assistant 是一个面向客服场景的 RAG 应用。它把售后政策、物流规则、账号 FAQ 和质保说明构造成知识库，用户提问时先检索相关文档片段，再调用 DeepSeek Chat API 生成带引用依据的回答。除了问答，它还支持客服对话质检，可以识别过度承诺、事实错误、遗漏信息和高风险回复，并通过内置评估集统计关键词命中率、引用命中率、拒答准确率和平均延迟。

这个项目的重点不是做一个聊天页面，而是把 LLM、检索、引用、质检和评估串成一个完整的 AI 工程闭环。

## 3 分钟技术讲解

系统分成四层：

1. 数据层：`data/knowledge_base` 存放 Markdown 知识库，模拟真实企业客服政策文档。
2. 检索层：`rag/loader.py` 加载文档，`rag/splitter.py` 切分 chunk，`rag/retriever.py` 做本地检索，`rag/chroma_retriever.py` 提供可选 ChromaDB 后端。
3. 生成层：`rag/generator.py` 将用户问题和 Top-K 检索片段拼成上下文，调用 DeepSeek / OpenAI-compatible API 生成回答。
4. 评估层：`qa/evaluator.py` 读取测试集，统计关键词命中率、引用命中率、拒答准确率和延迟。

客服质检模块在 `qa/quality_checker.py` 中实现。它会先检索相关政策，再让 LLM 按结构化字段输出质检分、风险等级、问题类型、合规判断、原因和建议回复。

## RAG 是怎么做的

流程是：

```text
用户问题
  -> 文档加载
  -> chunking
  -> Top-K 检索
  -> 构造上下文 prompt
  -> LLM 生成答案
  -> 返回答案和引用来源
```

核心设计点：

- 回答只基于检索到的知识库片段。
- 如果检索分数太低，系统会拒答或转人工。
- 页面展示来源文档和 score，方便检查模型是否有依据。
- 保留本地 fallback，即使 API 不可用也能演示核心流程。

## 如何减少幻觉

我做了四层约束：

1. 检索约束：先从知识库召回相关片段，不让模型凭空回答。
2. Prompt 约束：系统提示要求只基于知识库回答。
3. 阈值约束：如果最高检索分数低于阈值，系统拒绝回答。
4. 引用约束：每次回答展示来源文档和相似度分数，方便人工审查。

## 为什么做自动化评估

LLM 应用不能只看一次演示结果，需要持续评估。项目里用 `data/eval_questions.json` 构造测试集，评估：

- 关键词命中率：答案是否覆盖关键政策点。
- 引用命中率：是否找到了相关知识库来源。
- 拒答准确率：无关问题是否能拒绝回答。
- 平均延迟：真实 API 调用下的响应速度。

这让项目从 demo 变成了可迭代的 AI 系统。

## 面试高频问答

### Q1: 为什么不用直接让 LLM 回答？

直接让 LLM 回答容易产生幻觉，也无法保证回答符合企业政策。RAG 可以把回答限制在知识库范围内，并提供引用来源，适合客服、法务、售后等要求可追溯的场景。

### Q2: 这个项目和普通 Chatbot 有什么区别？

普通 Chatbot 主要是对话体验，这个项目更强调业务闭环：知识库检索、引用溯源、客服质检、结构化输出和自动化评估。它更接近真实公司内部 AI 工具。

### Q3: 如果上线真实公司，你会怎么优化？

我会优先做五件事：

- 使用真实 embedding 模型和 ChromaDB/pgvector。
- 增加 rerank，提高长文档检索准确率。
- 接入日志和用户反馈，统计命中率、延迟和成本。
- 建立更大的人工标注评估集。
- 加权限管理，区分客服、质检员和管理员。

### Q4: DeepSeek API 在哪里用？

DeepSeek Chat API 用在两个地方：知识库问答生成和客服质检报告生成。检索部分仍在本地完成，这样可以降低成本，也方便保留 fallback 机制。

### Q5: 你在项目中主要负责什么？

我负责从 0 到 1 设计和实现整个系统，包括项目选题、RAG 流程、知识库样例、客服质检 prompt、评估指标、Streamlit 页面、DeepSeek API 接入、README 展示和 GitHub 项目包装。
